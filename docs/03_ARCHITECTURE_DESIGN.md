# 03 アーキテクチャと設計

## 方針

MVP は Manifest V3 の Service Worker、Side Panel、Options、Chrome Storage で構成する。ブックマーク分類と整理計画生成は Chrome API から切り離し、Node.js 自動テストで検証できるようにする。

## 構成

| 層 | パス | 責務 |
| --- | --- | --- |
| Extension Shell | `extension/manifest.json` | 権限、画面、Service Worker 登録 |
| Background | `extension/background/service-worker.js` | Chrome API 操作、メッセージ処理、リンク確認、削除、適用、復元 |
| Shared Core | `extension/src/shared/` | ブックマーク整形、分類、分類先プリセット、整理計画生成、バックアップ整形、リンク判定、ルール候補生成 |
| Storage Adapter | `extension/src/storage/` | `chrome.storage.local` の読み書き |
| UI | `extension/sidepanel/`、`extension/options/`、`extension/popup/` | 操作画面と状態表示 |
| Tests | `tests/` | Chrome API に依存しないロジックと manifest の検証 |

## 実装選択肢

| 選択肢 | 内容 | 利点 | 欠点 |
| --- | --- | --- | --- |
| A | Vanilla JavaScript + MV3 静的ファイル | 依存が少なく、Chrome へ直接読み込める | 大規模化すると UI 部品の整理が必要 |
| B | React + Vite | UI 状態管理と部品化がしやすい | ビルド工程と出力設定が増える |
| C | 外部 API / LLM 分類前提 | 高度な分類候補を作りやすい | 権限、プライバシー、コスト、審査説明が重くなる |

## 採用判断

MVP は選択肢 A を採用する。理由は、今回の中核価値が「安全なブックマーク整理」であり、最初に検証すべきリスクは UI の装飾ではなく、分類理由、差分プレビュー、復元可能性だからである。Shared Core を分離しておくことで、将来 React + Vite へ移す場合も分類ロジックを再利用できる。

## UI 設計

- Side Panel は「スキャン」「分類結果」「埋め込みルール編集」「変更プレビュー」「リンク切れチェック」「危険操作」の順に並べる。
- Side Panel の分類結果セクションには、同一画面内で Options を iframe 表示する導線と、従来どおり別タブの Options を開く導線を併置する。
- リンク切れチェックは変更プレビューの後に置き、HTTP 404/410 の削除候補だけを表示する。
- Options は分類先設定、サイト割り当て、ルール管理、JSON 直接編集を同居させる。
- Options の JSON 編集欄は保存前の編集状態のソースとして扱い、有効な JSON ルールはモード切替前に `state.rules` へ同期して分類先設定、サイト割り当て、ルール管理を再描画する。
- 分類先設定では、大項目と中項目を既存分類先から選択でき、`新規作成...` を選ぶと新規入力欄を表示する。
- 分類先設定のフォームは、1 行目を大項目と中項目、2 行目をアイコン、軸、優先度、表示名、3 行目を理由、任意のサイト URL、認識ドメイン、追加操作として分ける。URL は必須にせず、必須項目と任意項目をバッジで示す。
- 分類先設定には追加済みの大項目・中項目の管理カードを置き、編集または削除できる。
- サイト割り当てでは、左に大項目と中項目のドロップ先、右に現在のブックマークから抽出したホスト名、ドメイン名、favicon サムネイル付きのサイト一覧を配置する。
- サイト割り当てには分類済みサイトの非表示切替を置き、長いホスト名や代表タイトルは省略表示する。favicon を取得できないサイトはドメイン名から生成した頭文字タイルで表示する。
- サイト割り当てで作成するのは分類ルールだけとし、ブックマーク本体の移動は Side Panel の既存プレビュー適用フローに限定する。
- 候補作成は、現在ブックマーク由来のローカル推定と、拡張機能内に持つ主要サイトの静的ルールバンクを組み合わせる。
- 候補作成の軸は UI で選択し、Shared Core 側で候補元と軸をフィルタリングする。
- ルール管理は分類先ごとのグループ表示、全体を俯瞰するグリッド表示、密度の高いリスト表示を切り替えられる。
- 分類先には組み込みアイコン、短いカスタム文字列、画像 URL サムネイルを持たせ、候補カードとルールカードの両方で表示する。
- 現在のブックマークから候補ルールを作る処理はローカルの `chrome.bookmarks` 読み取りと Shared Core の推定ロジックだけで完結させる。
- Popup は短い状態要約と導線に限定し、長い処理を置かない。
- 危険操作ボタンは通常操作と視覚的に分け、確認ダイアログを必須にする。
- Options の全ルール削除は、初期ルール復元とは別の危険操作として表示し、確認後に編集中ルール配列を空にする。

## 責務分割と設計パターン

- Shared Core は Chrome API と DOM に依存しない純粋関数群として維持し、分類、候補生成、リンク判定、整理計画を UI から分離する。
- Background は Service Worker のメッセージハンドラを Application Service として扱い、Chrome API 操作、Storage Adapter 呼び出し、Shared Core 呼び出しの調停に限定する。
- Options は画面モードごとの表示状態を `MODE_CONFIG` で管理し、モード追加時にラベル、同期要否、表示対象を局所的に増やせるようにする。
- ルール管理は `RULE_VIEW_RENDERERS` を Strategy として扱い、グループ、グリッド、リストの表示ロジックを分岐の塊ではなく描画関数単位で差し替える。
- Side Panel はユーザー操作ごとのコマンドハンドラを持ち、スキャン、リンク確認、バックアップ、適用、復元、Options 埋め込み表示の状態更新を明示的に分ける。
- DOM 生成では既存の状態オブジェクトを入力にして再描画する方針を取り、Chrome Storage への永続化は保存操作時だけに限定する。

## 安全設計

- 実行前に `ReorganizationPlan` を作り、UI で確認する。
- 整理先は `分類済みブックマーク / 大項目 / 中項目` を基本とし、既存ルールに大項目がない場合は Shared Core で補完する。
- リンク切れ削除は候補表示と確認ダイアログを必須にし、通信エラーや一時的な HTTP エラーを削除対象にしない。
- `整理 / 削除予定` は特別な分類先として扱い、整理計画では移動ではなく削除アクションに変換する。削除は Side Panel の整理適用確認を通った場合だけ実行する。
- 空フォルダ削除は、移動後に空になった元フォルダと整理済みブックマーク配下の空フォルダに限定し、Chrome のルートフォルダと整理済みブックマークのルートは保護する。
- 適用時に `rollbackSnapshot` を保存する。
- 復元は直前ジョブ単位で行う。
- 手動テスト用のバックアップは直前ジョブとは別に `bookmarkBackup` として保存し、必要な時点へ戻せるようにする。
- バックアップ復元は保存済みブックマーク ID の移動に限定し、ユーザーが後から追加したブックマークや空フォルダは削除しない。
- 分類、候補生成、整理、復元のためにブックマークの完全 URL やタイトルを外部 API へ送信しない。favicon サムネイル表示ではドメイン名だけを外部 favicon サービスへ送信する場合がある。
